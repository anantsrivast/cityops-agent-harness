# CityOps Agent Harness — Design

**Status:** Draft v1 (approved in brainstorming 2026-07-21; expected to iterate)
**Repo name:** `cityops-agent-harness` · **Python package:** `cityops_harness`

## What this is

A hands-on notebook series that teaches developers agent memory **and harness
engineering**: the sequel to the CityOps Copilot workshop
(`notebook_complete_ollama.ipynb`, which stays untouched as the prerequisite).
Learners take the copilot they built there and give it a self-improving
harness — tools, skills, scheduled promotion, context engineering — with every
piece of durable state living in Oracle Autonomous Database 23ai.

The capabilities are ported from `total_recall_complete (1).ipynb`, but **not
as-is**: the design review in `memory_promotion_review.md` identified serious
long-horizon gaps in both memory-promotion paths ("promotion without curation,
success without verification, growth without forgetting"). This implementation
closes those gaps, and the eval suite measures exactly what the review says the
original evals were silent on.

## Decisions made up front

| Decision | Choice |
|---|---|
| Database | OCI Autonomous Database 23ai (Always Free). Connected from Codespaces via mTLS wallet (base64 secret) or TLS connect string. |
| Embeddings | In-database ONNX (`DBMS_VECTOR`), loaded idempotently by the setup notebook. No embedding API keys, no text egress, regardless of LLM provider. |
| LLM providers | **All three first-class**: Ollama (local, tool-capable model pinned, e.g. `qwen3`), Anthropic, OpenAI. Selected by `LLM_PROVIDER` env var through one factory. |
| Observability | Langfuse, **both modes**: cloud (keys in Codespaces secrets) and self-hosted (docker-compose in devcontainer), selected by `LANGFUSE_MODE=cloud|local|off`. |
| Runtime | GitHub Codespaces (devcontainer). Also runs locally. |
| Format | 4 teaching notebooks + setup notebook. Each authored as `*_complete.ipynb` (source of truth); learner `*_todo.ipynb` generated by a build script that strips marked solution blocks — the two can never drift. |
| Notebook independence | Every notebook opens with a state check that verifies the DB artifacts the previous notebook should have left, with a one-cell backfill — independently runnable, but continuity across notebooks is itself the lesson. |

## The four notebooks

### 01 — Self-improving copilot *(closes the review's Path-2 gaps)*

The copilot's capabilities become registered tools (`log_finding`,
`find_similar_findings`, `report_event`, asset lookup) in a tool registry
retrieved by meaning; an agent loop calls them; recurring workflows are
harvested into SHA-versioned `SKILL.md` skills with two-level retrieval.
The gap fixes are the teaching beats:

- **Full-trajectory capture** (vs. "sorted set of tool names"): ordered tool
  calls with arguments and truncated results, so skill distillation is grounded
  in what actually ran. The notebook shows a distilled skill checked against
  its source trajectory.
- **Verified success** (vs. `success = bool(final)`): an LLM-judge scores each
  workflow outcome against the domain record (does the logged finding match the
  narrative? do cited priors exist?). Only verified outcomes feed
  `successes`. Verification scores are logged to Langfuse from this notebook
  onward.
- **Skill lifecycle** (vs. one-way promotion): skills are born *provisional*
  (lower prompt authority, labelled as such), carry a `skill_id` that later
  workflow captures link back to, and are promoted to *approved* or demoted to
  *retired* on accumulated linked outcomes. Staleness: skills record the schema
  SHA they were distilled against.
- **Retrieval thresholds** (vs. unconditional top-k): every skill/tool
  injection applies a distance cutoff — irrelevant queries inject nothing.
- **Merge hygiene** (vs. brittle intent dedup at 0.15): borderline embedding
  matches in a gray band get an LLM same-task confirmation before merge.

### 02 — Scheduled briefings *(closes the review's Path-1 gaps)*

The scratch→long-term promotion pipeline plus DBMS_SCHEDULER automations,
curated from the start:

- **Opt-in promotion with a triage gate** (vs. "everything graduates"): only
  designated paths (e.g. `/inbox`) are candidates; a cheap LLM triage
  classifies each note keep / discard / hold before staging. Working files,
  abandoned drafts, and offloaded tool dumps never reach long-term memory.
- **Provenance + supersession** (vs. "nothing forgets"): promoted memories
  record source file and revision; new facts supersede old ones (domain demo: a
  repaired defect supersedes the original finding); recall weights recency.
- **Whole-note promotion** (vs. blind 200-word chunk re-extraction): curated
  notes promote as units with provenance.
- **A really-scheduled consumer** (vs. manual-only drain): producer stays an
  in-DB DBMS_SCHEDULER job; the consumer also runs on a schedule, with a
  queue-depth health check so a stalled pipeline is visible, not silent.
- The payoff automation: a scheduled morning **asset-risk briefing** built from
  properly curated memory. Continuity-of-work (resuming the briefing plan in a
  new session) is demonstrated here.

### 03 — Context engineering

Compaction and offloading, measuring **fidelity, not just size** (the review's
critique of 8.1/8.2b):

- Compaction over a long simulated inspection season, charting card size vs.
  transcript size **plus** a card-correctness probe: does the card still
  contain the fact planted 20 turns ago when it is finally needed?
- Offloading of large tool results with the agent in the loop: the model must
  notice the reference, decide to fetch, and use the payload — not merely
  "bytes are recoverable."
- Offloaded blobs are explicitly excluded from promotion (closing the
  unmeasured Path-1 × offloading interaction the review flags).

As implemented (2026-07-22), the card is **additive** — the model may add or
correct a fact but never delete one. That is what makes the fidelity probe
trustworthy, and it means the notebook demonstrates no compression at all. See
*Deferred: strength-based card consolidation* below for the design that
resolves this; it is not built.

### 04 — Evals: five, through Langfuse

All evals run as Langfuse datasets with scored traces, across whichever
providers are configured. They target exactly the review's "silent on" column:

1. **Promotion precision** — labeled scratch corpus (signal + junk): does the
   triage gate graduate the right notes? Precision/recall on long-term
   memory admissions.
2. **Skill fidelity** — are distilled `SKILL.md` steps faithful to their source
   trajectories? LLM-judge comparing trajectory vs. skill.
3. **Skills-help** — the missing experiment: agent *with* harvested skills vs.
   *without*, on held-out task variants. Does learning improve task success?
4. **Card fidelity over long horizon** — multi-topic, adversarially spaced
   session; planted facts probed many turns later. Measures drift, not
   character count.
5. **Cost-per-correct-answer** — card vs. full-transcript vs. stateless, with
   repeated trials and variance (extends the original 8.4 beyond n=1).

## Deferred: strength-based card consolidation

**Status:** designed, not implemented. Revisit after phase 04. Nothing below
exists in code.

### Why

Notebook 03's live run showed the compaction card was **never smaller than the
transcript it stood for** — a flat ~0.71× from the first fold onward:

| card version | turns | transcript chars | card chars | ratio |
|---|---|---|---|---|
| v1 | 7 | 2,042 | 2,888 | 0.71× |
| v3 | 23 | 6,383 | 8,660 | 0.74× |
| v6 | 40 | 11,890 | 16,726 | 0.71× |

Two independent causes, and only the second is about forgetting:

1. **Nothing asks for compression.** v1 holds 7 facts drawn from 7 turns, at
   close to original length plus JSON scaffolding. The fold trigger fires on
   *pending turn* size and never inspects the card, so the model transcribes
   into a schema rather than summarising. A card with no ceiling has no reason
   to compress.
2. **Nothing may leave.** `merge_card` is additive by design, so there is no
   path back down once the card is oversized.

The obvious fix — evict by age, TTL-style — is wrong for this domain and wrong
about memory generally. Human forgetting tracks *retrieval strength*, not
arrival time: facts that keep being used survive, and the ones that fade lose
their **specifics before their gist**. In an inspection season the turn-4 fact
("Pier 3 bearing plate corroded, the reason Harbor Bridge was prioritised") is
precisely the one that must outlive turn 30's routine observation.

### Two triggers, kept separate

- **Fold trigger** (exists): pending turns ≥ `CARD_BUDGET` (2,000 chars) → one
  LLM call folds them in. Governs *when we summarise new material*.
- **Consolidation trigger** (new): immediately after every fold-and-merge,
  deterministic, no LLM —
  `while len(render_card(card)) > CARD_CEILING and len(card["facts"]) > FLOOR:
  evict the weakest fact`. Governs *when we forget*. Proposed
  `CARD_CEILING = 4_000` chars (~⅓ of the season's transcript, so it bites from
  fold 2), `FLOOR = 3` facts so a pathological run cannot strip the card bare.
  Re-checks after each eviction rather than evicting a fixed count.

### Strength model

```
strength = (1 + reinforcements) * 0.5 ** ((current_turn - last_seen_turn) / half_life_turns)
```

`half_life_turns = 8.0` (the season folds roughly every 7 turns). This is the
same half-life shape as `rerank_by_recency` in `promote.py`, deliberately, so
the codebase carries one decay model rather than two.

`reinforcements` comes from a new `reinforced: [fact_ids]` field on the
compaction call — which existing facts the new turns touched. Those facts get
`reinforcements += 1` and `last_seen_turn = current_turn`; a model that omits
the field degrades the policy to pure recency rather than failing. The
deterministic alternative (embedding similarity between new turns and existing
facts, using the in-DB embedder) costs a round trip per turn and is cruder.

### Consolidation, not deletion

Evicted facts collapse into a third top-level card key, `gist`, retaining their
ids:

```json
{"gist": [{"ids": ["f3", "f7"],
           "text": "Several earlier corrosion findings on Harbor Bridge piers."}]}
```

Retaining ids is what makes three-way scoring possible. `gist` lives inside
`CARD_JSON`, so `HARNESS_CARD` needs **no schema migration**.

### What this buys: a better metric

Fidelity stops being hit/miss and becomes **verbatim / gist-only / lost**. That
distinction is the whole point — a fact surviving as gist is a pass for "which
bridge was the problem?" and a fail for "which pier?", and the probe should say
which. Eval 4 in notebook 04 scores this directly.

### Shape of the work

- `context.py` (pure, unit-tested): `fact_strength(...)`,
  `select_for_eviction(card, current_turn, ceiling_chars=4_000, floor_facts=3,
  half_life_turns=8.0) -> (kept, evicted)`, `fact_status(planted_ids, card)`.
  `select_for_eviction` picks victims but does not write gist text.
- Notebook: the gist *text* is an LLM call (one per consolidation event,
  batching that event's evictions), keeping the repo's pure-logic/notebook-side-
  effects split. The compaction prompt gains the `reinforced` field.
- `card_fidelity` keeps its existing signature so current tests stay green; its
  `recall` becomes verbatim-recall, with `fact_status` carrying the nuance.
- The payoff cell runs the **same season under both policies**, additive and
  strength-based, plotting two curves against their fidelity.

### Known risks

- ~11 extra LLM calls per run (≈5 gist + 6 second-policy folds).
- At a 4,000-char ceiling an unreinforced planted fact may legitimately end up
  gist-only or lost. That is a true result, not a regression — so the assertion
  must be written against `fact_status` (e.g. "no planted fact is fully lost")
  rather than demanding verbatim survival, or the notebook goes red for the
  right reason.
- Retrieval-induced strengthening (the probe itself reinforcing what it asks
  about) is deliberately out of scope here; it belongs with eval 4.
